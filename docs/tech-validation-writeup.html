Architecture cheat-sheet

Before the scenario-by-scenario walk, three cross-cutting things the reviewer will see repeatedly:

1. The Schwab-Client-CorrelId header

It is a request header, not a query parameter. Every call our client makes includes it automatically — generated fresh per request as str(uuid.uuid4()) inside _get_headers() (src/schwab_advisor/client.py:92–94). That is why the CorrelId does not appear in any URL in the walkthrough output: 

Schwab-Client-CorrelId: 46c8fc9b-e2d8-425d-a647-d7d6c329b7d4
Schwab-Resource-Version: 1
Authorization: Bearer <access_token>

For the error demo (AL-0013, ST-0004) we thread a correl_id="" kwarg through so the header’s value is an empty string, which Schwab rejects.

2. Why “status feed” endpoints are POSTs

This recurs in your STFDPOST-* comments. Schwab models a status feed as a create a query-result resource pattern:

  1. POST /status-feed with a filter body creates a feed, returns a feed_id, and inlines the first slice of statusObjects in the response.
  2. Later, GET /status-feed/{feed_id} returns the same cached slice, and GET /status-feed/{feed_id}/status-objects/{obj_id}/status-events drills into one object’s history.

It is a POST because the filter body (accounts, categories, date range, etc.) can be much larger than a URL query string would comfortably carry, and because the resource that comes back is a persistent handle you can re-fetch. Semantically it is closer to “run this query, give me a pointer” than to “insert new data”. The Excel calls every one of these scenarios “Verify the user is able to send the field: X” — so Schwab’s acceptance bar is “does the POST accept field X and respond 200”, not “does the response prove the filter narrowed anything”.

3. OAuth & token lifecycle

Local ~/.schwab_tokens_alerts.json holds access_token, refresh_token, expires_at. The fly.io server schwab-oauth.fly.dev hosts /oauth/callback (the only redirect URI Schwab will accept for this app) and exposes /oauth/tokens so the local script can pull tokens down after a browser login via scripts/oauth_login.py.

Required — RE-0001

RE-0001 Unique correlator ID per call

Excel scenario: Verify that calls come across with unique Correlator IDs.
Expected: A unique correlator ID is passed on each service call. (Integration Specialist: This can be found in Extended_Fields.ServiceInfo.InternalCorrId.)

What we do

c1 = client._get_headers()["Schwab-Client-CorrelId"]
c2 = client._get_headers()["Schwab-Client-CorrelId"]
c3 = client._get_headers()["Schwab-Client-CorrelId"]
check(
    expected="All three generated correlator IDs are distinct UUIDs",
    test="n_unique == 3 and uuid_len == 36",
    n_unique=len({c1, c2, c3}),
    uuid_len=len(c1),
)

Why this is the right test

We produce UUIDs via Python’s uuid.uuid4() — a random 128-bit identifier. Schwab’s OpenAPI spec declares the header as string($uuid), which UUID-v4 satisfies by definition. The collision probability is ~2-61 even at a billion calls per day, so “three back-to-back unique” is sufficient evidence without enumerating millions. We also assert uuid_len == 36 (the xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx canonical form) so the reviewer sees the format matches the spec.

Enrollment

ENR-0001 Consent & Grant

Excel expected: The participant should get an access token.

What we do

auth.get_access_token() → reads the local token file, returns the current access_token string. We assert the token length is ≥ 50 chars (real Schwab tokens are >70 chars; this is a sanity floor).

On camera the upstream flow — user visits /v1/oauth/authorize, Schwab login page appears, Schwab redirects to our fly.io callback, fly.io POSTs the auth code to /v1/oauth/token and stores tokens — is demonstrated by running scripts/oauth_login.py immediately before the walkthrough. This scenario verifies the outcome: a working access token ended up in our local store.

ENR-0002 Enrollment UI pop-up (manual demo)

Excel expected: The Schwab Advisor Center® enrollment UI is properly presented in a new window.

We mark this [SKIP] in the script because the requirement is visual — a UI page in a browser. During the recording we run scripts/oauth_login.py, which calls open <authorize_url> and triggers exactly that UI pop-up. The Schwab sandbox login page appearing on camera is the evidence.

ENR-0003 Successful enrollment with valid SAC user/pw/OTP

Excel expected: The service returns the appropriate error message and a similar message is displayed in the Consuming application.
“Error Code”: Description — “You have successfully acknowledged and agreed to the Schwab OpenView Gateway Advisor Enrollment Use Agreement.”

Your question: how is this different from ENR-0002, and is that “Error Code” really a success message?

Good catch. The “Error Code” column is mis-labeled in the spreadsheet — what appears there is actually Schwab’s confirmation text the user sees after successfully agreeing to the Advisor Enrollment Use Agreement page. ENR-0002 is about the pop-up appearing; ENR-0003 is about successfully completing the flow (username + password + optional OTP) and landing on the consent-accepted page. They’re sequential steps of the same user journey.

What we do

auth.load_tokens() → assert both access_token and refresh_token are present. If the browser flow in oauth_login.py had failed at the consent page, fly.io would not have received the authorization code and our local tokens would be absent or stale. The check passes if the entire end-to-end flow succeeded.

Gap you’re right to flag: we do not programmatically assert the confirmation-text string appears in the UI. That would require headless browser automation of the Schwab sandbox login page, which is out of scope for this first-pass walkthrough. For the recording we visually show the “Success!” page that fly.io renders after the token exchange.

ENR-0004 Refresh token

Excel expected: The user should be able to get a refresh token after the access token expires in 3600 seconds or if they manually update the token.

Your question: shouldn’t the new token differ from the old one? Do we need to wait for expiration?

The Excel explicitly allows the “manually update” path (the second clause), which is what we do: call refresh_tokens() directly without waiting 3600 s. Under Schwab’s OAuth 2.0 implementation a fresh call to POST /v1/oauth/token with grant_type=refresh_token always returns a new access_token value (it is a fresh JWT signed by Schwab) plus a rotated refresh_token and a new 3600-second TTL.

Proposed strengthening

You’re right that a stronger assertion would be “new token string ≠ old token string”. We can add that check — empirically it does differ — and remove the chance that a future Schwab change silently returns the same token. I have queued this as a refinement (see the Gaps section).

Alerts

AL-0001 List alerts

Call client.get_alerts(page_limit=5, show_account='Show'). Assert response is a list and contains 1–5 items. This is the canonical “smoke test” for the endpoint.

AL-0002 Schwab-Client-Ids filter

Excel: Verify that the user can pass the Schwab-Client-Ids filter: account, masterAccount.

Your question: is masterAccount a type of Schwab-Client-Id? Should we also show a more specific account?

Yes — the Schwab-Client-Ids header accepts either account=X, masterAccount=Y, or a combined form masterAccount=Y,account=X (no whitespace; we learned this the hard way in the sandbox). We try the masterAccount= form explicitly and then probe the account= form using any formatted_account we saw in a prior response.

Known limitation: sandbox alerts return empty formatted_account strings, so the account= leg cannot actually hit Schwab in this environment. We print “no sub-account available in sandbox alerts to demo account= form”. In production alerts we’ll have real account numbers and this leg will run.

AL-0003 filter[startDate]

Call with filter_start_date='2026-01-01', then assert all(a.created_date ≥ '2026-01-01' for a in alerts). This proves the filter actually restricts the result set rather than merely being accepted.

AL-0004 filter[endDate]

You’re right: calling with filter_end_date='2026-04-17' (today) is a weak demonstration because that’s the default — nothing is filtered out. A better test is a date a few months back, so the reviewer sees the filter genuinely reduced the result. Queued for refinement.

AL-0005 filter[type]

Pass filter_types=['UserAlert']. Assert every returned alert has alert_type == 'User Alert'. All three returned alerts match.

AL-0006 filter[status]

Pass filter_status=['New']. Assert every returned alert has status == 'New'. All three returned alerts match (you saw 94 in your manual exploration with a larger page_limit).

AL-0007 filter[isArchived]

Two back-to-back calls — filter_is_archived=False then True. Assert every alert matches the requested flag. Demonstrates the filter is additive (not just accepted).

AL-0008 page[cursor]

Page 1: get_alerts(page_limit=5) → note next_cursor. Page 2: get_alerts(page_limit=5, page_cursor=…). Assert the two page-1 / page-2 id sets are disjoint. Output prints both sets so the reviewer can eyeball non-overlap.

AL-0009 page[limit]

Excel expected: The service returns the number of records (maximum value: 500).
You’re right: asking for 500 and checking ≤ 500 doesn’t prove the client honors the limit. A better sequence is e.g. page_limit=3 and assert len(alerts) == 3, then page_limit=500 for the max case. Queued for refinement.

AL-0010 sortBy

Your question: what are we actually testing?

Today we loop all 9 documented sort fields and assert each returns HTTP 200. That matches the Excel’s bar of “service accepts the field” but does not verify ordering for each field. AL-0011 covers ordering correctness for sortDirection by comparing Asc vs Desc of CreatedDate. Extending this to every field would require knowing the expected natural order per field (e.g. FormattedAccount string-sort vs Priority enum-sort), which isn’t spelled out anywhere.

AL-0011 sortDirection

Same call twice with Asc then Desc. Assert the first Asc item’s createdDate is less than or equal to the first Desc item’s. Stronger than “both accepted”.

AL-0012 showAccount (Mask/Show)

Two calls, Mask then Show. Assert '*' appears in the masked formatted_master_account but not in the shown one (e.g. '***4295' vs '8174295').

AL-0013 Bad correlator → 400

Call with correl_id="". The header goes out with an empty value; Schwab returns 400 with detail text “The Schwab-Client-CorrelId field is required.” We assert both the status code and the Excel-specified detail text appear in the response body.

AL-0014 Bad token → 401 SEC-0001

Instantiate a throwaway SchwabAdvisorClient(access_token= "not-a-real-token"), hit the same endpoint. Assert status == 401 and code == 'SEC-0001'.

AL-0015 Non-existent alert → 404

Call get_alert_detail(99999999, master_account='8174295'). Schwab returns 404 with a Schwab error code. We accept SEC-0002 (Excel) or SEC-0200 (sandbox returns the -0200 variant — a known Schwab discrepancy between doc and sandbox).

ALID-0001 Alert detail

Pull an id from get_alerts(page_limit=1), then get_alert_detail(id, master_account=…). Assert the returned detail’s id matches the requested id.

ALID-0002 showAccount on detail

Same id, two calls (Mask/Show), assert masked master contains '*' and shown master does not.

ALARCHPOST-0001 Archive a single alert

Pull one id, POST archive_alerts([id]). The body goes as {"alertIds": [N]}. The response contains an archiveDetails list with one entry per id in the request; we assert that list has exactly our id. Schwab’s warning “Please do not archive all accounts” is respected because we only ever pass a single id.

Status

ST-0001 Retrieve events from a status feed

Your question: how does this endpoint work? Multiple events per object?

Yes. A status feed (created by POSTing to /status-feed) contains a list of statusObjects — each one represents a case/envelope (e.g. “Digital Envelope / Account Open / AC Account Open”). Each statusObject has its own statusEvents list — the chronological history of that object (e.g. “New” → “Draft ready to send” → “In Progress”).

ST-0001 hits GET /status-feed/{feed}/status-objects/{obj}/status-events — it drills into one object’s event timeline. We verify the returned value is a list of events (may be empty for brand-new objects, which is valid per spec).

ST-0002 Pass feed_id in the status call

Calls GET /status-feed/{feed_id}, which returns the same inlined statusObjects the original POST returned. We assert len(get_response.status_objects) == len(post_response.status_objects). In other words: the feed is stable — hitting it twice yields the same set.

ST-0003 Pass object_id in the status call

Same GET as ST-0001 but framed as a test of the object_id path segment. Assert every returned event has status_object_id == <the id we sent>.

ST-0004 ⋅ ST-0005 ⋅ ST-0006 Error paths

Same mechanics as AL-0013 / AL-0014 / AL-0015 but on the status endpoints. ST-0006 has a documented doc/sandbox mismatch: Excel wants 404 + SEC-0002, sandbox returns 400 “The value is not valid for Schwab-Resource-Version” when feed_id is bogus. We accept either 400 or 404 and note the discrepancy.

STFDPOST-0001 Create a status feed

POST /status-feed with {"status":["New"]}. Assert a feed_id comes back. This confirms the resource-creation step works. Subsequent STFDPOST-* scenarios add more fields to the body.

STFDPOST-0002 status field

Your question: shouldn’t status be one-of, not all-of?

Per Schwab’s OpenAPI, status is an array<string> with an enum constraint: it means “give me objects matching any of these values”. Passing all five at once is therefore a legal, documented query (“any of New, Resolved, Canceled, In Process, Action Needed”). The Excel title “Verify the user is able to send the field: status: Action Needed, Canceled, In Process, New, Resolved” reads to us as “demonstrate that all five valid enum values are accepted”, which a single call covers.

If you prefer stricter evidence that each value is individually valid, we could loop each of the five and assert 200 per value. That’s a 5-line change.

STFDPOST-0003 masterAccounts field

Body includes "masterAccounts":["8174295"]. Assert "8174295" appears among the returned formatted_master_account values and the response is non-empty.

STFDPOST-0004 accounts field

You’re right this is a weak test today. We pass accounts=["12345678"] (a placeholder) and accept the resulting 401 SEC-0100 as “server parsed and authorization-checked the field.” Better: pull a real authorized account from a prior status-feed response (status_objects[*].clientInfo.auxProfileId) and submit that so we get a 200 with matching data. Queued for refinement.

STFDPOST-0005 / 0006 startDate / endDate

Per the Excel (“returns status info according to the field”), we now just verify the POST accepts the field and returns 200. The full response body is printed (first 25 lines) so the reviewer can see the feed created. If you want a stronger demonstration that the filter narrowed the result, we can re-introduce an unfiltered baseline comparison — we had that pre-alignment and removed it to match the Excel’s strict bar.

STFDPOST-0007 timeFrame

Your question: what does this field even mean?

timeFrame selects which date column the startDate/endDate filters apply against:

  • CreatedDate (default) — filter by when the statusObject was first created.
  • LastUpdatedDate — filter by most-recent activity (useful for polling workflows: “show me anything that changed since last sync”).

Because we don’t combine it with a date filter in this scenario, the test’s bar is just “both enum values are accepted”. That matches the Excel.

STFDPOST-0008 categories

Your comment: we just get 200 with an empty statusObjects array.

Correct — the sandbox has no objects tagged “Move Money” (most sandbox data is “Digital Envelope / Account Open”). The 200 + empty array is Schwab’s “filter accepted, no matches” response, which is a legal outcome.

A more convincing test for the recording is to filter on a category that does have data — e.g. ['Digital Envelope'] — so the reviewer sees a non-empty narrowing. Easy one-line change.

STFDPOST-0009 / 0010 / 0011 myqCaseId / serviceRequestConfirmationID / actionCenterEnvelopeId

Your comment: sandbox returns empty, so it doesn’t feel like a real test.

These three are exact-match filters that scope the feed to a single known case/SR/envelope. The Excel example values (WI-123456, SR813637257, 842993565) don’t exist in sandbox, so we get 200 + empty array: field accepted, no matches. To make these visibly meaningful in the recording we could:

  • Query a broad feed first, pick one actionCenterEnvelopeId out of the response, then re-query with that exact id — should return that single object.
  • Same trick for the other two fields, using any statusEvent.sourceId or similar that the sandbox actually returns.

The current tests confirm the plumbing (the body serialization, the POST body camelCase, the 200 response) but not the semantic narrowing.

STFDPOST-0012 includeAllEvents

Two POSTs, one with includeAllEvents=True, one with False. For each we count total events across every statusObject. Assert n_true ≥ n_false. In the sandbox we observe 140 events vs 114, so the flag clearly surfaces more event history.

To your meta-question “why is this a POST not a GET with filters” — see the architecture cheat-sheet. All filter input to a status-feed is POST-body; the resulting feed handle is then read-by-GET.

STFDPOST-0013 firstPageOnly

Excel: “true, false (default: false) this indicated whether 1000 or 2000 events should be returned”. We count events in both responses and assert n_true ≤ 1000 and n_false ≤ 2000. Sandbox only has 114 events total, so both are far below the cap; we explicitly print that in the Verify block so the reviewer understands why the numbers are identical. In production this would show up as a meaningful 2× difference.

STFDPOST-0014 showAccount

Same Mask/Show comparison as AL-0012 but against formatted_master_account on status objects.

Known gaps & recommendations

These are the concrete refinements worth doing before the screen recording, in order of reviewer impact:

#ScenarioChange
1AL-0004 Use a past endDate (e.g. 2026-03-01) so the filter demonstrably reduces the result count vs the unfiltered baseline.
2AL-0009 Use page_limit=3 and assert len == 3; keep a second call with 500 to demonstrate the cap.
3ENR-0004 Also assert new_access_token != old_access_token.
4STFDPOST-0004 Harvest a real authorized account id from a prior status-feed response and pass it so we see 200 + matching data (instead of 401 SEC-0100).
5STFDPOST-0008 Filter on a category the sandbox actually has ("Digital Envelope") so the response is non-empty and the reviewer sees real narrowing.
6STFDPOST-0009/0010/0011 Harvest a real actionCenterEnvelopeId from a broad feed and re-query it; prove round-trip exact-match works instead of just 200 + empty.
7STFDPOST-0005/0006 Optional: re-add the filtered-vs-baseline count comparison (was present, removed for Excel-strict interpretation) if you want stronger visual evidence of the date filter actually doing work.
8AL-0010 Optional: for each of the 9 sortBy fields, also make an Asc+Desc pair and assert ordering flips. Turns a 9-field “accepted” check into a 9-field “applied correctly” check.
