1. What We Built & Why

This is a Python client library for the Schwab Advisor Services API. It provides a clean, typed interface for our internal tools to interact with Schwab's API endpoints. Our immediate use case is realtime alerts polling — monitoring account alerts, tracking service request status, and pulling status feed updates for our advisor workflows.

🎯
Goal: Build a reusable, well-tested Python module that wraps the Schwab AS API so our internal tools can consume it without dealing with authentication, pagination, or response parsing.
📦
Scope: Three API products — AS Alerts (4 endpoints), AS Service Request (2 endpoints), AS Status (4 endpoints) — plus the existing AS Account endpoints.
✅
Status: All endpoints implemented, sandbox-verified with 19 live integration tests passing (including 8 edge case tests), plus 64 unit tests. Ready for production promotion.

3. OAuth Flow

We implement the standard OAuth 2.0 Authorization Code flow with a few Schwab-specific adaptations:

Authorization

# Schwab's authorize URL is non-standard: only client_id and redirect_uri
# No response_type or scope parameter required
def get_authorization_url(self) -> str:
    params = {
        "client_id": self.client_id,
        "redirect_uri": self.redirect_uri,
    }
    return f"{self.authorize_url}?{urllib.parse.urlencode(params)}"

Token Exchange

# Token exchange uses Basic Auth (base64 client_id:client_secret)
headers = {
    "Authorization": self._get_basic_auth_header(),  # Basic base64(id:secret)
    "Content-Type": "application/x-www-form-urlencoded",
}
data = {
    "grant_type": "authorization_code",
    "code": authorization_code,
    "redirect_uri": self.redirect_uri,
}

Token Lifecycle

Key Behaviors

  • Access token TTL: 30 minutes (1800s) for AS Account app, 60 minutes (3600s) for Alerts app
  • Refresh token TTL: 7 days
  • Rotating refresh tokens: Each refresh returns a new refresh token — the old one is invalidated. We persist immediately after every refresh.
  • Auto-refresh: get_access_token() automatically refreshes if expired
  • Token file permissions: Saved with 0o600 (owner read/write only)

4. API Client Design

Multi-Segment URL Routing

A key discovery during sandbox testing: each API product uses a different base path segment. AS Account uses bulk, while Alerts, Service Request, and Status use accounts.

# Each method specifies which segment it targets
_API_SEGMENTS = {
    "sandbox": "https://sandbox.schwabapi.com/as-integration/{segment}/v2",
    "production": "https://api.schwabapi.com/as-integration/{segment}/v2",
}

# Example: alerts use the "accounts" segment
response = self._request("GET", "/alerts", params=params, segment="accounts")

# Example: account profiles use the "bulk" segment (default)
response = self._request("GET", "/account-profiles", params=params)

Request Headers

Every request includes Schwab's required headers. Critically, Content-Type is only set on requests with a body — sending it on GET requests causes a 400 error.

def _get_headers(self, has_body=False, extra_headers=None):
    headers = {
        "Authorization": f"Bearer {self._get_access_token()}",
        "Schwab-Client-CorrelId": str(uuid.uuid4()),  # Unique per request
        "Schwab-Resource-Version": "1",
        "Accept": "application/vnd.api+json",
    }
    if has_body:
        headers["Content-Type"] = "application/json"
    return headers

Timeouts & Pagination

All requests use explicit timeouts (10s connect, 30s read) to prevent indefinite hangs. Default page size is 500 — the maximum the API will accept without silently returning empty results.

Body Format Inconsistencies

Important: Not All Endpoints Use JSON:API Format

The Schwab API is inconsistent in its request body formats across endpoints. We handle each one according to what actually works in the sandbox:

  • Alert archive: Flat body — {"alertIds": [15157526]}
  • Alert update: JSON:API wrapped — {"data": {"type": "alert", ...}}
  • Service request create: PascalCase flat body — {"TopicName": "...", "MasterAccount": "..."}
  • Status feed create: PascalCase flat body — {"Status": ["New"], "ShowAccount": "Mask"}
  • Status events post: camelCase flat body — {"myqCaseId": "...", "masterAccount": "..."}

6. Data Models

Every API response is parsed into typed Python dataclasses. This gives us autocompletion, type checking, and a clear contract for what fields are available.

Design Pattern

@dataclass
class Alert:
    id: int | str = ""
    formatted_master_account: str = ""
    alert_type: str = ""
    subject: str = ""
    status: str = ""
    # ... 15+ more fields ...
    raw_data: dict | None = None  # Full JSON for debugging

    @classmethod
    def from_dict(cls, data: dict) -> "Alert":
        attrs = data.get("attributes", data)  # Handles both JSON:API and flat
        return cls(...)

Why This Pattern?

  • raw_data field: Every model stores the original JSON. This is invaluable during development — when the API returns fields we haven't mapped yet, they're still accessible.
  • Default values on all fields: The API doesn't guarantee every field is present. Defaults prevent crashes on missing fields.
  • attrs = data.get("attributes", data): Handles both JSON:API wrapped responses (with id/type/attributes) and flat responses interchangeably.
  • Shared field parser: Alert and AlertDetail share 20+ fields. We extract a _parse_alert_attrs() helper to avoid duplication.

Model Hierarchy

7. Security Considerations

Credential Management

  • No hardcoded secrets: All credentials come from environment variables (SCHWAB_CLIENT_ID, SCHWAB_CLIENT_SECRET)
  • Token file permissions: Tokens are saved with chmod 0o600 (owner-only access)
  • Separate token files per app: Different Schwab apps use different token files to avoid cross-contamination
  • Basic Auth for token exchange: Client credentials are base64-encoded per OAuth 2.0 spec, not passed as query parameters

API Security

  • Unique Correlation ID per request: Every API call gets a fresh UUID for Schwab-Client-CorrelId, enabling request tracing
  • HMAC-protected OAuth server endpoints: The /oauth/start and /oauth/tokens endpoints on our callback server are protected with timing-safe HMAC comparison
  • XSS protection: The OAuth callback HTML response uses html.escape() on any error messages
  • HTTPS only: All Schwab API communication is over TLS

9. Deployment

OAuth Callback Server

We run a lightweight FastAPI server on Fly.io to handle the OAuth callback. This is needed because Schwab requires pre-registered callback URLs.

# server.py - Four routes

GET  /oauth/callback  # Receives OAuth callback, exchanges code for tokens
GET  /oauth/status    # Returns current token state (public)
GET  /oauth/tokens    # Exports tokens for local dev sync (API key protected)

10. Questions & Feedback for Schwab

During sandbox integration, we encountered several behaviors that required trial-and-error to resolve. We'd appreciate clarification on whether these are intentional or whether documentation could be improved.

1. Different Base URLs Per API Product (Undocumented)

AS Account endpoints use /as-integration/bulk/v2 while AS Alerts, Service Request, and Status use /as-integration/accounts/v2. This mapping is not documented in the API reference. We discovered it by extracting the base URL from the developer portal's sample curl commands.

Ask: Is there a reference that maps API product to base path segment? Will this change when moving to production?

2. Inconsistent Request Body Formats

Each endpoint uses a different body format, making it difficult to build a consistent client:

EndpointFormatExample
PATCH /alerts/{id}JSON:API{"data": {"type": "alert", "attributes": {...}}}
POST /alerts/archiveFlat{"alertIds": [15157526]}
POST /service-requestsPascalCase flat{"TopicName": "...", "MasterAccount": "..."}
POST /status-feedPascalCase flat{"Status": ["New"], "ShowAccount": "Mask"}
POST /status-eventscamelCase flat{"myqCaseId": "...", "masterAccount": "..."}

Ask: Is there a pattern for which endpoints use JSON:API vs flat format? Is the casing difference (PascalCase vs camelCase) intentional?

3. Content-Type Header Causes 400 on GET Requests

Sending Content-Type: application/json on a GET request (which has no body) returns "Request sent with Content-Type application/json failed payload validation." Most API gateways ignore Content-Type on bodyless requests. This required us to conditionally omit the header based on HTTP method.

Ask: Is this intentional behavior? It's unusual for an API gateway to reject a GET based on Content-Type.

4. GET /service-requests Returns Topic Catalog, Not Service Requests

The GET /service-requests endpoint returns a catalog of available topics and subtopics (e.g. "Money Movement" → "Wire", "Check Request"). It does not return a list of previously submitted service requests. The endpoint name suggests retrieval of existing SRs.

Ask: Is there a separate endpoint to list submitted service requests and their statuses? Or is the Status Feed the intended way to track SR progress?

5. Schwab-Client-Ids Header Required Only for Alert Detail

GET /alerts/detail/{id} requires the Schwab-Client-Ids: masterAccount=XXXXXXX header while GET /alerts does not. The alert list response includes the formattedMasterAccount field, so callers can pass it to the detail endpoint — but only if they know to use showAccount=Show on the list call first.

Ask: Could the detail endpoint accept the alert ID alone (since it's already scoped to our authorized accounts)? Or is this an intentional secondary authorization check?

6. POST /status-feed — Valid Status Values Not Documented

The Status field is required as a list, but valid values aren't documented. We discovered "New" and "Resolved" work through trial and error. Invalid values return "The value is not valid for status" without listing valid options.

Ask: What is the complete list of valid Status values? Could the error response include the allowed values?

7. POST /status-events Requires myqCaseId

Posting a status event requires a myqCaseId (max 16 characters) that references an existing open case. There's no documented way to create a new case via the API, and the ID format/source isn't clear.

Ask: What system generates myqCaseId values? Is this the MyQ case management system? How do we obtain a valid case ID programmatically?

8. page[limit] > 500 Silently Returns 0 Results

Requesting page[limit]=501 or higher on GET /alerts returns an empty data array with no error, no warning, and no meta.count. The max appears to be exactly 500. This is a dangerous silent failure — a client using a default limit of 1000 (a reasonable assumption) would believe there are no alerts.

Ask: Could limits above the max either clamp to 500, or return a 400 error? Is 500 documented as the max anywhere?

9. GET /status-feed/{feed_id} Does Not Support Pagination

Despite the response schema including Paging and Count models, passing page[limit] or page[cursor] to GET /status-feed/{feed_id} causes it to return 0 results. Without these parameters, it returns all objects (100+ in sandbox). The POST response also returns all objects inline.

Ask: Is pagination planned for this endpoint? For accounts with thousands of status objects, returning all at once could be problematic.

10. Service Request Attachment Format Unknown

We were unable to successfully submit a service request with an attachment. We tried JSON body fields (Attachments, Files, FileData) in both PascalCase and camelCase with base64-encoded content, as well as multipart/form-data (which is rejected as an invalid Content-Type). The API always responds with "An attachment is required."

Ask: What is the correct field name and format for file attachments on POST /service-requests? An example request body would be very helpful.

11. Error Code Reference

We catalogued these error codes through testing. A published reference would help integrators handle errors correctly:

CodeHTTP StatusMeaning
SEC-0001401Unauthorized (invalid or expired token)
SEC-0002404Endpoint not found (wrong base URL or not provisioned)
SEC-0200404Resource not found (valid endpoint, invalid resource ID)

Ask: Is there a complete error code reference? The distinction between SEC-0002 and SEC-0200 is useful for debugging but not documented.

12. Accept Header Must Be application/vnd.api+json

Using the standard Accept: application/json header works for some endpoints but fails silently for others (returning data in unexpected formats). The required header is application/vnd.api+json, which isn't prominently documented.

Ask: Could this be documented in the "Required Headers" section of the API reference?
